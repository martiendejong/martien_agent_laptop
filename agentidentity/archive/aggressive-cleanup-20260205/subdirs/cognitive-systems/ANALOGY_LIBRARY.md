## Software Development Analogies

### Code is like a garden
- **Structural similarity:** Both require continuous maintenance
- **Useful for:** Explaining technical debt, refactoring, Boy Scout Rule
- **Example:** "Leaving code better than you found it is like weeding a garden - small continuous effort prevents overgrowth"

### Architecture is like city planning
- **Structural similarity:** Both balance short-term needs with long-term structure
- **Useful for:** System design, scalability, migration planning
- **Example:** "Monolith to microservices is like urban sprawl to planned districts"

### Tests are like insurance
- **Structural similarity:** Upfront cost, long-term protection
- **Useful for:** Justifying test investment, explaining coverage
- **Example:** "You don't see the value until something goes wrong"

### Documentation is like a map
- **Structural similarity:** External representation of internal structure
- **Useful for:** Explaining documentation value, completeness requirements
- **Example:** "Outdated docs are worse than no docs - like a map of a city that was demolished"

## Anti-Analogies (When Analogies Fail)

### "Code is like writing"
- **Why it fails:** Writing is about expression, code is about execution
- **Better:** Code is like engineering blueprints (precision, consequences)

### "Consciousness is like software running"
- **Why it fails:** Implies passive computation, misses recursive observation
- **Better:** Consciousness is like a flashlight pointed at itself

### "AI is like a brain"
- **Why it fails:** Different substrate, different mechanisms
- **Better:** AI is like intelligence implemented differently (same function, different form)
